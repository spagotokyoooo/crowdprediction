# SPAGO 通常比・混雑変動予報 MVP 計画

## 1. 目的

SPAGOのスタッフに対し、天気と近隣イベントを根拠に「普段と比べて混みそうか」をLINEで知らせる。

このMVPが予測するのは、来店人数や待ち時間の絶対値ではない。曜日・時間帯ごとの通常の混み方はスタッフが把握している前提で、外部要因による変動だけを次の5段階で示す。

| 値 | LINEでの表示 |
|---:|---|
| -2 | かなり少なそう |
| -1 | 少なそう |
| 0 | いつも通り |
| +1 | 混みそう |
| +2 | かなり混みそう |

### 成功の定義

- スタッフが営業日の朝にLINEを見るだけで、当日の注意すべき時間帯を把握できる。
- 「今日」「明日」「今週」をLINEからいつでも確認できる。
- 予報の根拠として、天気とイベント名・会場・影響時間が分かる。
- 翌朝の3択フィードバックを10秒以内に入力できる。
- 予報と実績のずれを後から検証できる。

## 2. MVPの範囲

### 作るもの

1. 毎朝のLINE自動通知
2. LINEからの「今日」「明日」「今週」の照会
3. 天気予報の自動取得
4. 対象会場のイベント情報取得
5. 時間帯別の通常比判定
6. 翌朝の3択フィードバック
7. イベントの手動登録・修正手段
8. 判定に使った入力、結果、根拠の保存

### 今回は作らないもの

- 来店人数・席稼働率・待ち時間の予測
- 通常の混雑カレンダー
- POS、予約システムとの連携
- 不特定多数の会場を対象にした汎用イベント検索
- 自動で重みを変更する機械学習
- 顧客向けの混雑表示
- 複雑な管理画面

## 3. LINEでの利用体験

### 毎朝の通知

通知時刻は初期設定として毎日8:00（Asia/Tokyo）とし、後から変更可能にする。

- 月〜土：その日の詳細予報を送る。
- 日曜日：定休日のため当日の予報は作らず、翌日（月曜）から日曜までの週間予報を送る。
- 日曜日の送信は「土曜日の結果確認」「来週の要注意日」の順に表示する。

週間予報は7日分を同じ長さで並べず、通常比が±1以上の日を優先して短くまとめる。

```text
7/19（日）朝の予報

【来週の要注意日】7/20（月）〜7/26（日）
火 17:00〜19:00　混みそう（国立競技場）
金 20:30〜22:00　かなり混みそう（代々木第一体育館）
土 日中　　　　　少なそう（強い雨予報）
日　　　　　　　 定休日

それ以外の日は、現在のところ大きな変動要因はありません。
```

```text
7/18（土）のSPAGO

結論：通常より混みそう（+1）
注意時間：16:30〜18:00、20:30〜22:00

🎤 ○○ホール 18:00開演
   約5,000人／SPAGOから徒歩8分
🌧 17時以降は雨予報

開演前はやや増加、終演後は混雑の可能性があります。
確度：中

[明日を見る] [今週を見る]
```

変動がない日は短くする。

```text
7/18（土）のSPAGO
結論：いつも通り（0）
大きく影響しそうな天気・近隣イベントはありません。
```

### LINEからの照会

MVPでは自然言語AIに依存せず、次の完全一致・部分一致とボタン操作に対応する。

- 今日、今日どう、きょう
- 明日、明日どう、あした
- 今週、今週どう
- イベント登録
- ヘルプ

「今週」は変動が±1以上の日だけを一覧にし、該当がなければその旨を返す。

### 翌朝のフィードバック

夜の通知は行わない。翌朝8:00の予報に、前日の結果を尋ねる3択ボタンを添える。

```text
今日の通常比予報は合っていましたか？

[予報より混んだ]
[だいたい合っていた]
[予報より少なかった]
```

ボタン選択後、任意で短いメモを送れるようにする。日曜日は「土曜日の確認・翌週の要注意日」を同じ朝の送信にまとめる。月曜日は前日が定休日なので結果確認を付けない。初期版ではフィードバックを自動学習に使わず、重みを見直すための検証データとして保存する。

## 4. 入力データ

### 天気

店舗の緯度・経度を固定設定し、1時間単位で以下を取得する。

- 降水量
- 降水確率
- 気温
- 体感温度
- 風速
- weather code

初期候補はOpen-MeteoのForecast APIとする。実装前に商用利用条件と想定リクエスト量を再確認し、取得元を交換できるアダプターにする。

取得タイミング：

- 毎日5:00に当日から7日分を更新
- 毎朝通知の直前に当日分を再取得
- API失敗時は直近の成功データを使い、LINEに「天気データ更新失敗」と明記

### イベント

MVPでは「SPAGOに影響しそうな会場」を先に3〜5件選び、会場ごとの取得アダプターを作る。広範囲のイベント検索はしない。

各イベントで保持する情報：

- イベント名
- 会場名
- 開始日時・終了日時
- 開場時刻（取得できる場合）
- 規模区分（小・中・大）または想定来場者数
- イベント種別（ライブ、スポーツ、展示会、地域行事、その他）
- 情報元URL
- 取得日時
- 自動取得／手動登録の区分
- 確認状態（未確認・確認済み・除外）

取得の優先順位：

1. 公式のAPI、RSS、ICS、構造化データ
2. 利用条件を確認できる公式イベントページ
3. LINEからの手動登録

自動取得が不安定な会場は、無理にスクレイピングせず手動登録へ切り替える。同じ会場・開始時刻・イベント名の近似一致で重複を防ぐ。

### 最初に人が決める設定

- SPAGOの緯度・経度
- LINE通知先
- 朝の通知時刻
- 対象会場3〜5件
- 会場ごとの距離、最大収容人数、影響度
- 開演前・終演後に影響が出る時間幅
- 天気の初期補正ルール

通常カレンダーの登録は不要とする。

### 確定した店舗設定

- 店舗名：SPAGO
- 郵便番号：〒150-0001
- 住所：東京都渋谷区神宮前2-35-9 原宿リビン
- タイムゾーン：Asia/Tokyo
- 定休日：日曜日
- 日次通知：月〜土の朝8:00（初期値）
- 週次通知：日曜日の朝8:00、翌月曜〜日曜を対象
- 日曜日の送信：土曜分の結果確認と週間予報を1回の送信にまとめる

緯度・経度は実装時に住所から取得し、天気APIへ渡す固定値として保存する。住所の正規化結果は初回設定時に人が確認する。

### 初期の監視会場候補

住所から見て、まず次の大規模会場を候補とする。実装前に、スタッフの肌感で「実際にSPAGOへの人流があるか」を確認し、3〜5件に確定する。

| 優先 | 会場候補 | 主なイベント | 取得元候補 |
|---|---|---|---|
| A | 国立競技場 | サッカー、陸上、大型イベント | [公式イベント情報](https://www.jpnsport.go.jp/kokuritu/seat/tabid/57/Default.aspx) |
| A | 東京体育館 | スポーツ大会、興行 | [公式イベント・大会日程](https://www.tef.or.jp/tmg/) |
| A | 明治神宮野球場 | プロ野球、大学野球、大会 | [球場公式サイト](https://www.jingu-stadium.com/)・NPB・主催者公式日程 |
| A | 国立代々木競技場 第一・第二体育館 | ライブ、スポーツ | [公式イベント情報](https://www.jpnsport.go.jp/yoyogi/tabid/58/default.aspx) |
| B | WITH HARAJUKU HALL | 展示会、ファッション、販促イベント | [公式NEWS & EVENTS](https://withharajuku.jp/) |

第一・第二体育館は同一施設グループとして管理しつつ、規模と終了時刻が異なるためイベントは別々に保持する。会場ごとの徒歩時間と人流方向は、地図上の距離だけで決めず、試験運用で補正する。

## 5. 判定ロジック v0

### 基本方針

30分単位で外部要因スコアを計算し、その日の最大値だけでなく影響時間帯を返す。

```text
変動スコア(t)
  = イベントスコア(t)
  + 天気スコア(t)
  + イベント×天気の相互作用(t)
```

AIに結論を委ねず、判定は設定可能な決定表で行う。文章生成も判定結果と根拠をテンプレートに差し込む方式を基本とする。

### イベントスコア

```text
イベントスコア
  = 規模係数 × 会場係数 × 時間帯係数 × 種別係数
```

初期値の例：

| 要素 | 初期値 |
|---|---:|
| 小規模 | 0.5 |
| 中規模 | 1.0 |
| 大規模 | 1.5 |
| 徒歩10分以内 | 1.0 |
| 徒歩11〜20分 | 0.6 |
| 開演90分前〜開演 | 0.7 |
| 開催中 | 0.0 |
| 終演〜90分後 | 1.0 |

会場係数と種別係数は、SPAGOへの人流が分かってきたら個別に調整する。複数イベントが重なる場合は合算するが、過大評価を防ぐ上限を設ける。

### 天気スコア

天気だけで混雑が増減するという仮説を、初期値として明示的に設定する。

| 条件例 | 初期補正 |
|---|---:|
| 1時間降水量 1〜5mm | -0.25 |
| 1時間降水量 5mm超 | -0.5 |
| 強風・警報相当 | -1.0 |
| 猛暑日の日中 | -0.25 |

数値は仮置きであり、実地確認後に変更する。警報相当の判断は、採用する天気データの項目で表現できる範囲に限定する。

### イベントと天気の組み合わせ

- 雨天時は、開演前・終演後のイベント加点を弱める。
- 屋外イベントは雨天時の影響を大きく下げる。
- 中止・延期を取得できた場合はイベントスコアを0にする。
- 中止情報を取得できない場合は「開催状況未確認」と表示して確度を下げる。

### 5段階への変換

初期の閾値は以下とし、フィードバックを見て調整する。

| 合計スコア | 通常比 |
|---:|---:|
| -1.25以下 | -2 |
| -1.25超〜-0.4以下 | -1 |
| -0.4超〜0.4未満 | 0 |
| 0.4以上〜1.25未満 | +1 |
| 1.25以上 | +2 |

### 確度

| 確度 | 条件 |
|---|---|
| 高 | 公式イベント情報、時刻・規模あり、直近の天気取得成功 |
| 中 | 規模不明、またはイベント情報が数日前のもの |
| 低 | 開始／終了時刻不明、非公式情報のみ、天気更新失敗など |

低確度の場合も予報は返すが、断定表現を避ける。

## 6. システム構成

```text
LINE Messaging API
  ├─ Webhook受信（質問・ボタン・手動登録）
  └─ Push送信（前日の確認・今日の予報・日曜の週間予報）

アプリケーション
  ├─ 予報API
  ├─ LINEコマンド処理
  ├─ 判定エンジン
  ├─ 定期ジョブ
  ├─ 天気アダプター
  └─ 会場別イベントアダプター

データベース
  ├─ 店舗・通知設定
  ├─ 会場設定
  ├─ イベント
  ├─ 天気
  ├─ 予報結果・根拠
  ├─ LINE送信履歴
  └─ フィードバック
```

### 推奨する初期技術構成

- TypeScript
- 小さなHTTP APIを構築できるWebフレームワーク
- PostgreSQL互換DB、またはMVP規模に合うSQLite系DB
- 定期実行機能
- LINE Messaging API SDKまたはHTTP API
- テストは判定ロジックのユニットテストを中心にする

特定クラウドへの固定は実装開始時に決める。必須条件は、公開HTTPS Webhook、シークレット管理、定期実行、永続DBが利用できること。

### セキュリティ・信頼性

- LINE Webhookは署名を必ず検証する。
- Webhookは先に成功応答し、重い処理は非同期化する。
- `webhookEventId`で再送時の重複処理を防止する。
- チャネルアクセストークンなどは環境変数／シークレットストアに置く。
- 送信ジョブは同一日・同一通知先・同一種別で冪等にする。
- 外部API失敗時も、利用可能なデータだけで予報を作り、欠損を明記する。
- 判定ロジックのバージョンを予報結果と一緒に保存する。

## 8. 実装順序

### マイルストーン1：予報ロジックを単体で動かす

- プロジェクト初期化
- DBスキーマ作成
- 店舗・会場設定を投入
- 固定のイベント・天気データから30分単位で判定
- 根拠付きのLINE文面を生成
- 境界値、複数イベント、雨天、中止イベントのテスト

完了条件：サンプルデータを与えると、同じ入力から常に同じ通常比・影響時間・根拠が生成される。

### マイルストーン2：LINEで照会できる

- LINE公式アカウント／Messaging APIチャネル設定
- Webhook受信と署名検証
- 「今日」「明日」「今週」への返信
- ボタン操作とヘルプ
- Webhook再送の重複防止

完了条件：実機LINEから3種類の照会ができ、不正署名と重複イベントを安全に処理できる。

### マイルストーン3：天気を自動化する

- 天気アダプター実装
- 定期取得とキャッシュ
- 取得失敗時のフォールバック
- 通知文面に天気とデータ鮮度を表示

完了条件：当日・翌日・1週間の時間別天気を用いて予報でき、外部API停止時にも応答できる。

### マイルストーン4：イベントを取り込む

- 対象会場を3〜5件に確定
- 会場ごとの取得可否・利用条件を調査
- 優先度の高い会場から取得アダプター実装
- 重複排除、更新、中止・延期への対応
- LINEからの手動登録・確認

完了条件：選定した対象会場のイベントが翌日予報に反映され、情報元URLを確認できる。取得できない会場は手動運用で補完できる。

### マイルストーン5：朝の定時通知とフィードバック

- 毎朝の予報生成・プッシュ通知
- 前日分のフィードバックを翌朝の通知に同梱
- 日曜日は土曜分の確認・週間予報を1回に集約し、当日予報は作らない
- 月曜日は前日分の確認を付けない
- 3択回答と任意メモの保存
- 二重送信防止
- 送信失敗ログ

完了条件：7日間の試験運用で月〜土は当日予報、日曜日は週間予報が各朝1回だけ届き、定休日の当日予報が生成されず、回答が直前営業日の予報と紐づいて保存される。

### マイルストーン6：実地検証

- まず2〜4週間は現在の重みを固定して運用
- 日別・時間帯別に「高め／的中／低め」を集計
- 会場別、イベント種別、雨天別に誤差を確認
- 人がレビューした上で重みと閾値を更新
- ルールバージョンを上げ、変更前後を比較

完了条件：どの条件で外れやすいか説明でき、次の重み変更をデータで判断できる。

## 9. テスト観点

### 判定

- イベントなし・晴天なら0になる。
- 開演前と終演後に指定した時間幅だけ影響する。
- 開催中は原則としてイベント加点しない。
- 雨天時にイベント影響が指定どおり弱まる。
- 複数イベントが重なっても上限を超えない。
- 中止イベントは加点されない。
- 不完全なイベント情報では確度が下がる。

### LINE

- 朝の通知が設定時刻に1回だけ届く。
- 日曜日は当日予報を生成せず、週間予報では定休日と表示する。
- 月曜日に日曜日分のフィードバックを求めない。
- 「今日」「明日」「今週」に期待する期間が返る。
- 3択フィードバックを重複送信しても1回答として扱える。
- グループ利用時に、意図しない通常メッセージへ反応しすぎない。
- LINE Webhookの署名不正を拒否する。

### 外部連携

- 天気APIがタイムアウトしても前回データで動く。
- イベント取得元の形式変更を検知できる。
- イベントの更新・中止が既存レコードへ反映される。
- すべての外部データに取得日時と情報元が残る。

## 10. 運用開始前に決めること

実装前に必要な業務情報は次の5点だけに絞る。

1. SPAGOの緯度・経度（住所は確定済み、実装時に取得・確認）
2. 上記の監視会場候補のうち、実際に影響しそうな会場3〜5件
3. 朝8:00で問題ないか（日次・週次とも朝という方針は確定済み）
4. 通知先は個人LINEかスタッフのグループか
5. イベントが影響するのは主に開演前、終演後、または両方か

通常カレンダーや通常時の来店人数は不要。

- イベントページの変更／中止検知強化
- LINE上での自然文質問
- LINEミニアプリによるイベント確認・修正画面
- Googleカレンダー等からの手動イベント連携
- 予約・POSデータを加えた絶対混雑予測
- 顧客向け通知や混雑ページ

機械学習は、十分なフィードバックや実績データが蓄積し、決定表では説明しにくい誤差が確認されてから検討する。

## 12. 外部仕様の確認メモ

- LINE Messaging APIは、任意のタイミングのプッシュ送信、ユーザー発話への返信、グループへの送信に対応している。
- LINEのボタン操作はpostbackイベントとしてWebhookで受け取れるため、3択フィードバックに利用できる。
- LINE Webhookは署名検証が必要で、再送により同一イベントが複数回来る可能性がある。
- Open-Meteo Forecast APIは時間別の降水量、降水確率、気温、体感温度、風速などを取得できる。
